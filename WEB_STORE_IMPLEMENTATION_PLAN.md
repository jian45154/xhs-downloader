# 小红书视频转录扩展：从零实现与 Web Store 上架路径

## 目标

从零独立实现一个 Chrome 扩展，用于：

1. 在用户主动操作后识别当前小红书笔记中的视频。
2. 下载用户有权使用的视频。
3. 调用本机提取音频。
4. 调用本机转录服务生成 TXT、SRT 和 JSON。
5. 通过 Chrome Web Store 审核并公开发布。

原则：不复制现有扩展代码、图标、文案或品牌资产；权限最少；功能单一；数据默认留在本机。

## 推荐产品边界

首个上架版本只做：

* 识别当前打开的单篇视频笔记。

* 下载一个视频。

* 可选调用本机 FFmpeg 提取音频。

* 可选调用本机 Whisper 转录。

* 下载 ZIP，其中包含视频、笔记正文和转录文件。

首版不做：

* 批量抓取、账号自动化或绕过访问限制。

* 登录凭据、Cookie 或浏览器存储读取。

* 云端上传、用户账号、支付或订阅。

* 水印去除、版权保护规避或隐藏式下载。

* 自动启动外部程序。

这些限制可以降低版权、隐私、安全和商店审核风险。

## 架构

### Chrome 扩展

* Manifest V3。

* Content script 只负责读取当前页面可见内容和媒体地址。

* Service worker 负责获取媒体、调用本地服务和生成下载文件。

* 用户必须点击明确按钮后才开始处理。

建议权限：

```json
{
  "permissions": ["downloads", "activeTab", "scripting"],
  "optional_host_permissions": [
    "https://www.xiaohongshu.com/*",
    "https://*.xhscdn.com/*",
    "http://127.0.0.1:18765/*"
  ]
}
```

尽量在用户首次使用相应功能时请求可选权限，并解释用途。

### 本地转录伴侣

* 独立于扩展发布。

* 仅监听 `127.0.0.1`，不得监听局域网或公网地址。

* 使用随机安装令牌验证扩展请求，不能只依赖 CORS。

* 限制上传大小、并发数、文件类型和处理时长。

* 临时文件转录结束后立即删除。

* 不记录视频内容、转录正文或完整媒体 URL。

* 提供清晰的启动、停止和卸载方式。

推荐将 Python、faster-whisper 和模型封装成签名的 Windows 安装程序。扩展本身不得包含或远程执行 Python 代码。

## 阶段一：版权清理

1. 建立新的空仓库。
2. 不复制当前扩展的 JavaScript、HTML、CSS、图标或文案。
3. 根据公开网页行为重新编写功能。
4. 创建原创名称和视觉资产。
5. 避免名称或图标暗示它是小红书官方产品。
6. 在商店描述中注明与小红书不存在隶属、授权或背书关系。
7. 建立 `THIRD_PARTY_NOTICES.md`，记录所有依赖及许可证。

验收标准：

* 新仓库历史中没有原扩展文件。

* 所有图片和文案均为原创或具有明确商业授权。

* 每个第三方依赖均有可核验许可证。

## 阶段二：实现最小扩展

1. 创建 Manifest V3 基础项目。
2. 只在用户点击扩展按钮后注入解析脚本。
3. 从当前笔记 DOM 中读取：

   * 页面标题或作者名称；

   * 可见正文；

   * 当前视频元素的 `currentSrc`；

   * 必要时读取视频元素内的 `source`。
4. 不扫描浏览历史、其他标签页或后台网络请求。
5. 在下载前显示：

   * 即将处理的媒体数量；

   * 是否调用本地转录；

   * 内容权利确认提示。
6. 下载失败时显示真实错误，不得伪报成功。

不要把对页面私有数据结构或混淆脚本的逆向解析作为首版核心方案。DOM 变化时宁可提示暂不支持，也不要扩大抓取范围。

验收标准：

* 只对当前活动标签页生效。

* 没有用户点击时不下载、不转录。

* 不读取 Cookie、Local Storage 或账号凭据。

* 图片笔记和无视频页面不会误触发转录。

## 阶段三：实现本地转录伴侣

1. 使用 `faster-whisper` 的稳定版本。
2. 默认使用适合普通 CPU 的模型和量化设置。
3. 提供 `/health` 和 `/transcribe` 两个端点。
4. 首次配对时生成随机令牌：

   * 令牌由本地伴侣生成；

   * 用户明确复制或批准配对；

   * 扩展使用 `chrome.storage.local` 保存；

   * 每次请求通过专用请求头发送。
5. 请求必须同时满足：

   * 来源是已配对客户端；

   * 令牌正确；

   * Content-Type 和文件大小合法；

   * 当前并发量没有超限。
6. 返回 TXT、SRT 所需内容和结构化片段。

验收标准：

* 普通网页不能调用转录接口。

* 服务关闭时扩展给出清晰提示。

* 转录期间取消操作可以清理临时文件。

* 重启电脑后用户可以通过明确入口重新启动服务。

## 阶段四：隐私与安全

编写公开隐私政策，至少说明：

* 扩展读取哪些页面信息。

* 视频和文本为何被处理。

* 转录默认只在用户电脑本地完成。

* 是否存储任何数据以及保存期限。

* 不出售或共享用户数据。

* 用户如何删除本地数据并卸载伴侣程序。

* 联系方式和政策生效日期。

安全措施：

* 所有外部输入都视为不可信。

* 文件名必须清理，防止路径穿越。

* 不执行下载内容。

* 限制 ZIP 和媒体尺寸，避免内存耗尽。

* 不使用远程托管代码、`eval` 或动态脚本。

* 固定依赖版本并生成软件物料清单。

* 对安装程序进行恶意软件扫描和代码签名。

验收标准：

* Chrome 扩展源码中没有密钥。

* 没有遥测，或遥测为明确选择加入。

* 隐私政策与实际行为逐项一致。

* 删除扩展及伴侣后没有残留用户内容。

## 阶段五：商店材料

准备：

* 扩展名称、简短描述和详细描述。

* 原创的 128×128 图标。

* 至少一组真实功能截图。

* 隐私政策公开 URL。

* 支持邮箱和支持页面。

* 单一用途说明。

* 每项权限的必要性解释。

* 数据使用披露。

* 内容权利与非官方声明。

建议商店文案避免：

* “无限下载”“破解”“去限制”等表述。

* 宣称支持下载任何受版权保护内容。

* 暗示获得小红书官方授权。

* 无法证明的隐私、安全或性能承诺。

建议加入：

> 本工具仅用于保存你本人创作、已获授权或法律允许使用的内容。请遵守平台条款和适用法律。本产品与小红书无隶属或授权关系。

## 阶段六：审核前测试

测试矩阵：

* Windows 10 和 Windows 11。

* Chrome 当前稳定版。

* 服务已安装、未安装、未启动和转录失败。

* 图文笔记、普通视频笔记、长视频和无声视频。

* CDN 请求失败、页面结构变化和超大文件。

* 非小红书页面。

* 扩展更新和伴侣版本不兼容。

发布包检查：

* 删除日志、缓存、模型文件、测试视频和 `__pycache__`。

* ZIP 根目录直接包含 `manifest.json`。

* 不包含源码映射中的无关或上游代码。

* 不包含开发服务器地址和调试开关。

* 对发布 ZIP 进行一次全新 Chrome Profile 安装测试。

## 阶段七：提交与发布

1. 注册 Chrome Web Store 开发者账户并启用两步验证。
2. 支付一次性开发者注册费。
3. 上传扩展 ZIP。
4. 完成隐私、权限和数据使用表单。
5. 提交为非公开测试版本。
6. 用新用户环境完成安装与卸载测试。
7. 修正审核反馈。
8. 审核稳定后改为公开发布。

本地伴侣应通过独立官网或可信发布渠道提供，不要将可执行文件塞进扩展包。下载页应提供校验值、版本、许可证和卸载说明。

## 收费策略

首次上架建议完全免费，先验证审核稳定性和真实需求。不要在第一个版本同时引入支付、账号和云服务。

后续低风险收费路径：

1. 免费扩展保留基础下载功能。
2. 本地伴侣提供付费高级功能，例如批量队列、更大模型和字幕格式转换。
3. 使用外部网站完成购买和许可证管理。
4. 在扩展内清楚披露哪些功能收费。
5. 不将用户浏览数据用于广告或信用评估。

收费前需要补充：

* 服务条款。

* 退款政策。

* 税务和消费者保护合规。

* 许可证验证的离线容错。

* 不依赖隐蔽设备指纹的反滥用方案。

## 上架门槛

只有满足以下条件才提交审核：

* [ ] 代码和视觉资产均为独立实现。
* [ ] 第三方许可证完整。
* [ ] 权限已压缩到最小范围。
* [ ] 本地服务具备请求认证和资源限制。
* [ ] 隐私政策与实际行为一致。
* [ ] 不读取账号凭据或绕过访问控制。
* [ ] 版权提示和非官方声明清楚可见。
* [ ] 全新 Chrome Profile 测试通过。
* [ ] 发布包不含日志、缓存和开发文件。
* [ ] 支持、卸载和数据删除流程可用。

## 推荐执行顺序

1. 新建干净仓库并完成独立 UI 和图标。
2. 实现只处理当前页面的单视频下载。
3. 完成本地伴侣认证和安装程序。
4. 接入 TXT、SRT、JSON 转录结果。
5. 编写隐私政策、许可证和支持文档。
6. 完成安全测试和发布包清理。
7. 先以免费、非公开版本提交审核。
8. 审核稳定后公开发布，再评估收费。
